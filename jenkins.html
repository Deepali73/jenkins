Explanation:
Code ‚Üí GitHub ‚Üí Jenkins ‚Üí Docker Image ‚Üí Container ‚Üí Production üöÄ

Real-life analogy:

Let's break it down:

1. Developer pushes code to GitHub:
Think of GitHub as a digital cookbook. Developers write new recipes (code), save them in GitHub.

Jenkins is like a master chef always checking for new recipes. As soon as a new one is added, Jenkins jumps into action!

5. Docker runs Container ‚Üí Sent to Production:
Docker starts a container (the actual running dish), and it's delivered via Zomato-like production environments to your users.

üß† CMD vs ENTRYPOINT in Docker

ENTRYPOINT: Hard-coded behavior of a container (cannot be overridden easily)
CMD: Default arguments passed to ENTRYPOINT (can be overridden at runtime or build time)

Key Difference:
CMD can be overridden when you run the container using the Docker CLI (e.g., docker run)
ENTRYPOINT will always execute ‚Äî it ignores override arguments unless explicitly set to allow them.

Example:
ENTRYPOINT ["python3"]
CMD ["app.py"]
Runs: python3 app.py

Think of it like:
ENTRYPOINT = the oven set to "bake" (always fixed)
CMD = the timer or temperature you want (can be changed)
